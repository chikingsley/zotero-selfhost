# Cloudflare Production Runbook

This is the product/release path around the compatible API. It is not a list of
API compatibility gaps.

## Current Baseline

- Worker: `zotero`
- Custom domain: `zotero.peacockery.studio`
- D1 database: `zotero`
- R2 bucket: `zotero-attachments`
- Latest measured live API result: `451 passing`, `22 pending`, `0 failing`
- Real Zotero Desktop smoke: passing against the live Worker

## Recommended Cloudflare Shape

Keep this as a plain Workers project in `server/`:

- `wrangler.jsonc` declares Worker name, D1 binding, R2 binding, compatibility
  date, `nodejs_compat`, observability, and required secrets.
- `package.json` owns deployment scripts.
- `.env.example` lists local/deploy secret names without real values.
- `worker-configuration.d.ts` is generated by `wrangler types`.

Cloudflare's Deploy to Cloudflare button flow supports Workers apps, automatic
provisioning for D1 and R2 bindings from Wrangler config, and package-script
setup for D1 migrations.

Reference docs:

- Cloudflare Deploy buttons:
  <https://developers.cloudflare.com/workers/platform/deploy-buttons/>
- Workers best practices:
  <https://developers.cloudflare.com/workers/best-practices/workers-best-practices/>

## One-Click Deploy Path

The root README includes a Deploy to Cloudflare button pointed at the isolated
`server/` subdirectory:

<https://deploy.workers.cloudflare.com/?url=https://github.com/chikingsley/zotero-selfhost/tree/main/server>

Cloudflare requires the source repository to be public before other users can
use that button.

Recommended release path:

1. Make the repo public or publish a clean template repo.
2. Keep the Deploy to Cloudflare button pointed at the isolated `server/`
   subdirectory.
3. Keep private custom domains out of reusable template config; users can add
   their own custom domain after deploy.
4. Keep `server/package.json` scripts:
   - `deploy`: apply D1 migrations, then deploy Worker.
   - `deploy:worker`: deploy Worker only.
   - `db:migrations:apply`: apply D1 migrations through the `DB` binding.
5. Keep `server/package.json.cloudflare.bindings` descriptions so the deploy UI
   explains why D1, R2, and each secret exist.
6. Keep generated resource IDs out of reusable template docs; Cloudflare can
   rewrite IDs during provisioning.

## Backup And Restore

D1:

- Short-term recovery: use D1 Time Travel. Cloudflare documents it as always on
  for production-backend D1 databases and restorable to a point in time.
- Portable export: use `wrangler d1 export DB --remote --output=database.sql`.
- Restore/import: use `wrangler d1 execute DB --remote --file=database.sql`
  after reviewing the SQL and migration state.

R2:

- Use the S3-compatible API with an R2 access key.
- Configure the AWS CLI profile with region `auto`.
- Back up by syncing/listing the `zotero-attachments` bucket through the R2
  endpoint for the account.

Reference docs:

- D1 import/export:
  <https://developers.cloudflare.com/d1/best-practices/import-export-data/>
- D1 Time Travel:
  <https://developers.cloudflare.com/d1/reference/time-travel/>
- R2 with AWS CLI:
  <https://developers.cloudflare.com/r2/examples/aws/aws-cli/>

## Real Data Import

The Zotero.org API key is useful as an import credential, not as authentication
to this server.

Recommended importer shape:

1. Pull user library metadata from Zotero.org using `ZOTERO_IMPORT_API_KEY`.
2. Normalize into the same write paths the compatibility API uses.
3. Pull attachment files when Zotero.org permissions and storage URLs allow it.
4. Write into the self-hosted D1/R2 deployment.
5. Run `bun run smoke:desktop` against the target endpoint after import.

## Auth Recommendation

For the first real release, keep this single-owner:

- One generated admin username/password for setup/admin routes.
- One generated initial API key.
- User-created API keys after setup.
- No OAuth until there is a real multi-user product reason.

That keeps the Cloudflare deploy path simple and avoids building an account
system before the self-hosted API story has real users.

Client connection model:

1. Deploy the Worker and keep the generated Worker URL or attach a custom
   domain.
2. Create a local self-host API key with root credentials:
   `POST /users/1/keys`.
3. Configure Zotero Desktop or another compatible app with that API base URL
   and API key.

There is no automatic live link to Zotero.org. A Zotero.org API key is only for
an explicit import/copy command.
